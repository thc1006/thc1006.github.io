1. ORCA Hand 官方規格

參考文獻：arXiv 2504.04259

1.1 基本參數

  • 馬達：Pollen Robotics B1044 BLDC（最大扭矩 0.16 Nm，連續 0.08 Nm）。
  • 編碼器：MPS MA600（12-bit SPI 絕對值編碼器）。
  • 腱繩：Dyneema 0.5mm 釣魚線（強度 30kg+）。
  • 彈簧：拉伸彈簧（Restoring force）用於手指伸展回彈。

1.2 17個 Joint 的完整定義 (v2)

原版 ORCA 手掌擁有 17 個自由度，由 6 顆馬達透過滑輪與差速器驅動：

  • 拇指 (Thumb)：3 joints (Abduction, Proximal, Distal) - 需 2 Motors。
  • 食指 (Index)：3 joints - 1 Motor。
  • 中指 (Middle)：3 joints - 1 Motor。
  • 無名指 (Ring)：4 joints (包含掌骨彎曲) - 1 Motor (與小指偶合)。
  • 小指 (Pinky)：4 joints - 1 Motor (與無名指偶合)。

1.3 為何不能 1:1 復刻

硬體與環境限制說明：
  1. 馬達數量限制：原版需 6 顆馬達，而強茂提供給我們的配額僅為 5 顆 B1044。
  2. 驅動板體積限制：原版使用極小化的 Orbita 驅動板藏在手掌內。我們使用的是官方 KXB EVB 開發板，體積較大，必須外掛於前臂（Forearm）。
  3. 加工難度：原版的差速滑輪 (Differential pulleys) 需要高精度 SLA 或 CNC 加工，我們在賽場環境只能依賴 FDM 3D 列印。

2. 我們的簡化變形：5-DoF 欠驅動

2.1 動作取捨

為適應硬體限制，我們將機構改為 1 馬達對應 1 手指 的獨立欠驅動 (Underactuated) 設計：

手指 (Finger) 分配馬達 關節數 致動方式 (Actuation) 回彈方式 (Return)
Thumb (拇指)M12 (僅屈曲)單一腱繩貫穿彈簧 / 橡皮筋
Index (食指)M23單一腱繩貫穿彈簧 / 橡皮筋
Middle (中指)M33單一腱繩貫穿彈簧 / 橡皮筋
Ring (無名指)M43單一腱繩貫穿彈簧 / 橡皮筋
Pinky (小指)M53單一腱繩貫穿彈簧 / 橡皮筋
設計妥協 (Trade-off)

放棄了原版拇指的外展/內收自由度。雖然無法進行「精確捏取硬幣」等極度精細動作，但對於展示「抓握 (Grasp)」、「指引 (Point)」等大姿態已經非常足夠，符合語音控制的情境需求。

2.2 每根手指的機構細節

  • Spool (捲線軸)：直接鎖固在 B1044 的外轉子上。
  • Tendon Routing (走線)：Dyneema 線從 Spool 出發，經過前臂導環，進入掌心，穿過近端指骨、中端，最終固定在遠端指骨末端。
  • 動作原理：馬達收卷時拉動腱繩，造成手指彎曲；馬達放鬆時，由背側彈簧拉回使手指伸直。

2.3 力矩 Budget 驗證

根據 B1044 特性：連續扭矩 0.08 Nm。假設捲線軸半徑 r = 8mm (0.008m)：

  • 腱繩拉力 F = T / r = 0.08 / 0.008 = 10 N (~1 kgf)。
  • 對於單根手指的屈曲，10N 的拉力足以克服復歸彈簧的阻力並提供穩定的抓握力。峰值扭矩可短暫提供 ~2 kgf 的拉力應付突發阻力。

3. 完整電路設計

3.1 主控 ↔ 驅動板：PWM × 5 配線

PM32F407 需配置定時器，發送 5 組 PWM 訊號給 5 張 PM30225V-0401。

驅動板 (KXB EVB) 端接腳處理必看清單：
  • J7 (Sensorless Jumper)：必須拔除跳線帽！ 本案採用有感 FOC 控制。
  • J9 (PWM Mode)：必須插上跳線帽！（將 PWM_MODE 腳位拉高）。
  • P4 排針的 HIN1：此為 PWM 訊號輸入點。
  • 共地 (Common GND)：5 張驅動板的 GND 必須與 PM32F407 的 GND 確實相連，否則訊號會飄移。

* 註：詳細 PM30225V KXB EVB 實體 silkscreen 請參考硬體技術手冊。

3.2 馬達 ↔ 編碼器：Molex 12-pin FFC × 5

B1044 馬達尾部內建了 MPS MA600 磁感編碼器，並透過 12-pin 軟排線 (FFC) 引出。我們需要準備 FFC 轉接板，將訊號拆解：

  • 動力線 (U, V, W)：接回各別 KXB EVB 的馬達輸出端子。
  • SPI 訊號 + 電源 (3.3V, GND)：接回 KXB EVB 板上的 MCU。注意：FOC 閉迴路運算是在 PM30225 內部執行，主控只負責下達指令。

3.3 電源架構

為求簡潔，全系統採用單一 12V 輸入。

電流預算 (Current Budget)

組件電壓預估最大耗電流備註
PM32F407 EVB5V (降壓)~300 mA含 Codec 與周邊
B1044 + KXB EVB (單指)12V~600 mA峰值扭矩時可能微升
全系統總計 (5指同時作動)12V~3.5 A建議選用 12V 5A 電源供應器確保餘裕

PM32F407 EVB 自動 Power Path

PM32F407 主控板支援從 USB 或外部 5V 供電。實作上，可從其中一張 KXB 驅動板的 5V LDO 輸出端拉線至主控板的 5V 腳位，達成全系統僅需一個 12V 變壓器插頭的設計。

3.4 音訊路徑 (M2階段)

為了實現語音辨識與互動，我們直接利用 PM32F407 EVB 上的 WM8978 音訊 Codec。

  • 硬體介面：EVB 左下角具備 3.5mm MIC 與 EARPHONE 插孔。
  • 通訊協定：MCU 透過 I2C 設定暫存器，並透過 I2S 傳輸 PCM 音訊。

3.5 觸覺感測 (選配, M3後)

預計在指尖貼附 FSR 薄膜壓力感測器，並透過分壓電路將類比電壓回傳至主控 ADC，實現抓握力道的閉迴路控制。

5. 軟體 Toolchain 與編譯驗證

開發環境建置是初期的主要挑戰。我們採用 Keil uVision 5 搭配 ARM Compiler 6，並參考了開源社群 (SunboAZ) 的修正方案來解決原廠 SDK 的相容性問題。

Keil uVision 5 (AC6) 設定要點

在編譯原廠底層庫（如 system_hk32l08x.c）時，若遇到 inline 函數報錯，請採取以下修正：

// 將原本會報錯的宣告：
__inline void SystemCoreClockUpdate(void)

// 改為標準 C99 語法：
static inline void SystemCoreClockUpdate(void)

// 或者使用編譯器屬性強制 inline：
__attribute__((always_inline)) void SystemCoreClockUpdate(void)

6. 韌體架構與 FreeRTOS

為了確保音訊採樣與馬達控制的即時性，PM32F407 將運行 FreeRTOS 作業系統。任務規劃如下：

  • Audio Task (High Priority): 負責處理 I2S DMA 中斷，將 WM8978 收到的 PCM 音訊存入 Ping-Pong Buffer，並將處理完的音效輸出。
  • Motor Control Task (Real-time): 根據計算出的角度目標，更新 5 個 Timer 的 PWM 佔空比 (Duty Cycle)。
  • AI/Logic Task (Normal Priority): 處理語音辨識的輕量化特徵擷取（如 MFCC），或接收來自 UART/USB 的上位機指令，轉換為手部動作序列。

7. AI 模型與語音推理

考量到 PM32F407 (Cortex-M4) 的算力限制，我們不跑龐大的 LLM，而是採用 Keyword Spotting (KWS) 或輕量級指令辨識模型。

  • 工具：使用 TensorFlow Lite for Microcontrollers (TFLM) 或是 CMSIS-NN。
  • 指令集：訓練 5-10 個明確的關鍵字，例如："Open", "Close", "Point", "Thumb"。
  • 回饋機制：辨識成功後，不僅手部做出動作，WM8978 也會播放預載的「滴」聲或簡短的語音確認。

8. 預期風險與 Mitigation

潛在風險 (Risks) 緩解策略 (Mitigation)
FOC 驅動板通訊延遲
主控發送 5 路 PWM 若不同步，手部動作會怪異。		 硬體層面使用 MCU 的 Advanced Timer (TIM1/TIM8) 確保 PWM 同步輸出；軟體層面優化 FreeRTOS Task 排程。
腱繩斷裂或打結
Dyneema 線在滑輪處摩擦或纏繞。		 列印時確保導線孔(Routing holes) 平滑；軟體設定馬達轉角極限 (Software Limits)，防止過度收卷。
MCU 記憶體不足
音訊 Buffer + TFLM 模型吃光 RAM。		 精簡模型架構；降低音訊採樣率至 16kHz；若真不夠，退而求其次將 AI 運算移至 PC 端。
