§ 01 · Overview

Executive Summary

本文件是 MakeNTU 2026 強茂企業獎 Proposal A VoiceMotion 的電路、韌體、AI、機構一體設計規劃。目標：以語音指令直接驅動 ORCA-inspired 腱繩式靈巧手，完全基於強茂 PM32F407 主控、5 張 PM30225V-0401 FOC 驅動板、5 顆 B1044 BLDC 馬達與板上 WM8978 音訊 codec；前臂裝馬達、掌內放被動機構，5 條 Dyneema 腱繩穿腕軸承到指尖。

ORCA 原版 17-DoF 使用 17 顆 Feetech 伺服；本案硬體僅 5 顆 B1044，因此取 ORCA 機構設計哲學（腱繩路由、popping joints、被動復位）配合 5 自由度欠驅動簡化：每顆馬達透過單腱繩拉整指 MCP + PIP 合併彎曲，放棄外展與腕關節控制。

2.3 為何不能 1:1 復刻

硬體與環境限制
  1. 馬達數量限制：原版 17-DoF 需 17 顆伺服（或多顆馬達 + 差速器），我們的配額僅 5 顆 B1044。
  2. 驅動板體積：原版用極小化 Orbita 驅動藏在手掌內；我們用 KXB EVB 開發板，體積較大，必須外掛於前臂。
  3. 馬達類型：原版 Feetech 伺服內建位置環，我們 B1044 是 BLDC + 外掛 FOC 驅動，通訊模型完全不同（serial daisy-chain vs 5 獨立 PWM）。
  4. 加工限制：原版差速滑輪需 SLA / CNC，我們只能靠 FDM 3D 列印，必須避開高精度差速機構。
取捨原則

機構抄 ORCA，電子用自家 — 下載 orcahand_description 的 STL（v2/ 目錄），把 finger / palm 拿來印，拇指對掌角度固定在 palm geometry 內，不要外展馬達；電氣層走我們自己的 PM32F407 + PM30225V + B1044 架構。

§ 03 · Design

5-DoF 欠驅動簡化

為適應硬體限制，我們將機構改為 1 馬達對應 1 手指 的獨立欠驅動（Underactuated）設計：一條腱繩同時連結 MCP、PIP、DIP 三個關節；馬達拉繩 → 三關節依序彎曲；放繩 → 背側彈簧被動拉回。這是 RUKA Hand（arXiv 2504.13165）已驗證的低複雜度方案。

對應 Proposal A §2.5 KWS 詞彙表的 demo 動作：GRASP / RELEASE / POINT / STOP / DEMO。放棄拇指外展/內收意味無法做「精確捏取硬幣」等極細動作，但對「抓握」、「指引」等大姿態完全夠用。

3.2 每根手指的機構細節

  • Spool 捲線軸：3D 列印 PLA，直接鎖固在 B1044 輸出軸；Dyneema 纏 2-3 圈 + CA 膠一點固定。
  • Tendon Routing 走線：Dyneema 從 Spool 出發，經前臂導環 → 手腕軸承（608ZZ Ø 10 mm，降摩擦）→ 掌心 → 穿近端指骨 → 中端 → 固定在遠端指骨末端。
  • 動作原理：馬達收卷 → 腱繩拉 → 手指彎曲；馬達釋放 → 背側彈簧回拉 → 手指伸直。

§ 04 · Wiring

完整電路設計

本章按子系統分解：主控 → 驅動板 PWM、馬達 ↔ 編碼器 FFC、電源分配、WM8978 音訊、FSR 觸覺。每節含對應 diagram + 表格 + 實務警告。

4.1 主控 ↔ 驅動板：PWM × 5 配線

PM32F407 使用 TIM1 CH1-4（PE9/11/13/14）+ TIM3 CH1（PA6），輸出 5 組 20 kHz PWM 給 5 張 PM30225V-0401。每路杜邦線獨立 20 cm，共地必要。

馬達指位TIMMCU 腳AFKXB 接口
M1拇指 ThumbTIM1_CH1PE9AF1P1 Pin 3
M2食指 IndexTIM1_CH2PE11AF1P1 Pin 3
M3中指 MiddleTIM1_CH3PE13AF1P1 Pin 3
M4無名 RingTIM1_CH4PE14AF1P1 Pin 3
M5小指 PinkyTIM3_CH1PA6AF2P1 Pin 3
KXB EVB 跳線 + 共地必看清單
  • J7 Sensorless Jumper：必須拔除（本案採 sensored FOC，MA600 編碼器走 SPI）。
  • J9 PWM Mode：必須插上（拉高 PWM_MODE 腳位）。
  • P4 排針 HIN1：PWM 訊號輸入點。
  • 共地 (Common GND)：5 張驅動板 GND 必須與 PM32F407 GND 實連，否則 PWM duty 會飄移。
  • EMI：PWM 線避免與 12V 動力線捆綁；實測有雜訊可換 Ø22 AWG 絞線或屏蔽線。

4.2 馬達 ↔ 編碼器：Molex 12-pin FFC × 5

B1044 馬達尾部內建 MPS MA600 磁感編碼器，與 UVW 三相共用單一 12-pin FFC 排線（Molex 52745-1297, 0.5 mm pitch）。一條 FFC 同時承載相電流（Pin 1-6）+ VDD/GND（Pin 7-8）+ SPI（Pin 9-12）。

FFC 採購陷阱

0.5 mm pitch FFC 有 same-side 與 opposite-side 兩種排列。買錯 → 上電瞬間把 UVW 相線接到 SPI 邏輯腳 → 燒 MA600 編碼器。

SOP：收到馬達後微距拍攝兩端 Molex 金屬觸點方向 → 對照 KXB P2 方向 → 下單；建議 same + opposite 各備 3 條。

4.3 電源架構

全系統單一 12V 輸入：12V 5A DC PSU → 5-way 分電板 → 5 張 KXB VIN 並聯 + PM32F407 EVB JACK1（6-12V 範圍內），GND 共地。

電流預算

工況單張 KXB5 張合計12V PSU 需求
全部空轉0 A0 A< 1 A idle
全部輕負載0.3 A1.5 A≥ 2 A
一指輕握其他空1.2 + 4×0.11.6 A≥ 2 A
5 指同時堵轉（瞬間）1.8 A9 A⚠ 5A PSU 會跳電

保守結論：12V 5A PSU 夠用 demo，5 指同堵機率極低（KWS 指令一次只觸發一個動作）。想保險就改 12V 10A。

4.4 音訊路徑（M2 階段）

直接利用 PM32F407 EVB 上的 WM8978 音訊 Codec：MCU 透過 I²C1（PB8/PB9）設定 Codec 暫存器，透過 I²S2（PB12/13/14/15 + PC6 MCLK）傳輸 PCM 音訊。MIC_JACK / EAR_JACK 位於 EVB 左下角。

WM8978 I²C address = 0x1A（7-bit）/ 0x34（8-bit），與板上 EEPROM (0x50) 及相機 (0x21-0x30) 同匯流排，address 不衝突。PB14/PB15 衝突警告：同時是 USB OTG D-/D+，用 I²S2 時 USB OTG 不能啟動；我們 flash 走 CH340 USART（Type-C J8）不是 OTG，不衝突。

4.5 觸覺感測（選配，M3 後）

指尖貼附 FSR 薄膜壓力感測器 + 10kΩ 上拉分壓，類比電壓回傳至 MCU ADC，實現抓握力道閉迴路。

§ 05 · Toolchain

軟體 Toolchain 與編譯驗證

開發環境建置是初期主要挑戰。採用 Keil uVision 5.37 搭配 ARM Compiler 6.18（ARMCLANG），並整合開源社群（SunboAZ）的 AC6 兼容性修正 patch。AC5 已從 MDK 5.37 installer 移除，所有 vendor 的 .uvprojx 都必須翻 AC6 才能編譯。

5.1 Keil uVision 5 (AC6) 設定要點

編譯原廠底層（如 system_hk32l08x.c）時若遇 inline 函數報錯，採以下三種修正之一：

// 原本會在 AC6 下報錯的宣告：
__inline void SystemCoreClockUpdate(void)

// 改法 A — 標準 C99 語法：
static inline void SystemCoreClockUpdate(void)

// 改法 B — 編譯器屬性強制 inline：
__attribute__((always_inline)) void SystemCoreClockUpdate(void)

// 改法 C — 透過本 repo 的 patch script 自動修正：
$ ./patches/apply_vendor_patches.sh
// idempotent，可重複執行；修正 PUTCHAR_PROTOTYPE 巨集的 __GNUC__ guard
PUTCHAR_PROTOTYPE bug 背景

Vendor BSP 在 pm32f4xx_def.h 用 #ifdef __GNUC__ 判斷編譯器。但 ARMCLANG 為了 GCC 相容性也定義 __GNUC__，導致 AC6 下 printf 的 __io_putchar retarget 被誤當成 newlib 走向。patches/apply_vendor_patches.sh 修正這個條件為 #if defined(__GNUC__) && !defined(__ARMCC_VERSION)（與 ST HAL 2015+ / Geehy / CMSIS 5 同 pattern）。細節見 Issue #1；公開 PR 見 SunboAZ PR #1。

5.2 GCC arm-none-eabi 備援工具鏈

若 TFLM runtime + DS-CNN int8 model 總量超過 MDK-Lite 32 KB code 上限，改用 arm-none-eabi-gcc + 手寫 Makefile（SDD §8 已定）。關鍵旗標：

# CPU / FPU
CPU_FLAGS  = -mcpu=cortex-m4 -mthumb
FPU_FLAGS  = -mfpu=fpv4-sp-d16 -mfloat-abi=hard

# 最佳化（TFLM + CMSIS-NN 效能必要）
OPT_FLAGS  = -O2 -ffunction-sections -fdata-sections

# 預處理器
DEFS       = -DARM_MATH_CM4 -D__FPU_PRESENT=1 -DPM32F407xx

# 連結
LDFLAGS    = -Wl,--gc-sections -Wl,-Map=$(TARGET).map -T pm32f407_flash.ld -lm
§ 06 · Architecture

韌體架構 + FreeRTOS

為確保 I²S 音訊採樣與 5 路 PWM 馬達控制的即時性，PM32F407 運行 FreeRTOS 作業系統，將互相獨立的即時任務隔離到各自的 Task，透過 FreeRTOS 的 message queue + semaphore 做橋接。

§ 07 · AI

AI 模型與語音推理

考量 PM32F407 Cortex-M4 的算力限制，不跑 LLM，而採用 Keyword Spotting (KWS) 輕量語音指令辨識。模型走 int8 量化 → 透過 TFLM 或 CMSIS-NN 在板上推理 → 命中後手部動作 + WM8978 回播 ACK 音效確認。

7.1 KWS 規格

項目規格
輸入16 kHz PCM，25 ms frame，20 ms shift，40 mel bins，FFT 512，49 frames → [1, 40, 49, 1] int8
模型DS-CNN-S（int8 全量化）+ CMSIS-NN 加速
推理時間< 5 ms per frame（板上實測）
信心門檻0.85 — 低於視為 Unknown，不觸發動作
詞彙GRASP · RELEASE · POINT · STOP · DEMO（若準確率 < 90% 降為 3 詞）
板上精度目標≥ 95% validation set（push-to-talk 場景）

7.2 資料收集 SOP

資料集是 ML 模型的上限 — 越早錄越好。目標：5 詞 × 50 段 × 5 位錄音人 = 1,250 clips。

  1. 錄音環境：安靜室內，60 cm 距離，PM32F407 EVB 板上 MIC_JACK 直錄 + PC 錄音對照。
  2. 採樣規格：16 kHz / 16-bit mono / 1 秒 WAV per clip（前後留 100 ms 空白）。
  3. 每人錄音任務：5 詞 × 10 段（含語氣變化：正常、快、慢、小聲、大聲）× 5 次輪替 = 250 clips / 人。
  4. 前處理：scripts/training/prepare_dataset.py 做 DC removal、peak normalize、靜音段修剪。
  5. split：70% train / 15% val / 15% test，同一錄音人不跨 split。

7.3 訓練到板上推理

# 階段 1 — 本地 GPU 訓練 (一次性, ~1 小時 @ RTX 3060)
$ python scripts/training/train.py \
    --data ~/datasets/kws_panjit \
    --arch ds_cnn_s \
    --optimizer adamw --lr 1e-3 --scheduler cosine \

# 階段 2 — int8 量化 (post-training, representative dataset)

# 階段 3 — 嵌入為 C array (xxd)
$ xxd -i model_kws_int8.tflite > src/controller/ai/model_kws.c
$ sed -i 's/unsigned char/const uint8_t/g' src/controller/ai/model_kws.c

# 階段 4 — 板上推理一致性驗證
$ python scripts/training/verify_on_board.py \
    --port COM3 --samples ~/datasets/kws_panjit/test

7.4 回饋迴圈 — 不只動手，還要 WM8978 回應

辨識成功的多模態回饋

命中 KWS 詞之後，除了馬達動作，WM8978 同步播放預載的「滴」聲（200 ms 440 Hz 方波 pre-encoded WAV）或短語音「Okay」。這個設計的價值：

  1. 使用者確認：判官/觀眾知道系統有聽到（避免「機器沒動 = 沒聽到 還是 動作中」的歧義）。
  2. demo 節奏：聲音 ACK 讓 demo 像對話而非靜默指令，更吸引人。
  3. Debug 輔助：confidence 低時播放「！」短音，確認是「沒聽到」還是「聽到但沒信心」。

7.5 記憶體與降級方案

狀況應對
TFLM runtime 吃爆 Flash 或 SRAM改用 CMSIS-NN 純 int8 kernel（不需要 TFLM interpreter overhead）
DS-CNN-S 模型仍太大精簡為 2-layer TC-ResNet；降 mel bins 40 → 20；降 FFT 512 → 256
音訊採樣率佔 RAM16 kHz 改 8 kHz；1 秒 window 改 0.5 秒
上述都不夠退而求其次 — AI 運算移至 PC 端（USB 走 CDC 傳 PCM 過去），MCU 只做收音 + 動作

8.2 建構順序（MVP-first 6 階段）

  1. 階段 1 — 純 MCU + 音訊（今天可驗，不需手掌）· M0 / M2
    • ✅ D4 電源 LED 亮 + CH340 auto-ISP + keystone hello M0
    • □ 4 顆跳線帽 J88/J90/J93/J94 閉合
    • □ Keil build vendor I2S_Audio + apply_vendor_patches.sh → flash → UART banner
    • □ 改寫為 I²S full-duplex loopback → 麥克風 → 耳機 → M2 PASS
  2. 階段 2 — 單馬達閉環（1 顆 B1044 + 1 張 KXB）· M1
    • □ FFC 腳位驗證（目視 + 萬用表，不通電）
    • □ 12V PSU → 單張 KXB 上電驗無煙
    • □ Flash vendor PM30225V sample V 變體
    • □ PM32F407 PE9 PWM duty=30% → 馬達轉
    • □ 示波器量 UVW 三相波形（若有示波器）
  3. 階段 3 — 5 馬達同步（桌上 5 顆並排）
    • □ 12V PSU + 分電板 → 5 張 KXB 並聯
    • □ 5× PWM 同時輸出 duty=0 → 全靜止
    • □ 逐顆 duty=30% → 都轉；同步 duty=30% → 全轉（< 2 A）
  4. 階段 4 — ORCA 手掌 + 腱繩
    • □ 印 ORCA palm + 5 指（賽前完成至少 80%）
    • □ 組裝、穿腱繩、張力調整
    • □ 單指測試：手動拉 → 合攏 → 放 → 彈回
    • □ 馬達接腱繩 → duty=50% × 1 s → 指動 → M1 完成
  5. 階段 5 — 語音控制（M3-M7）
    • □ 錄 5 詞 × 50 段 → DS-CNN 訓練 → int8 量化 → 嵌入
    • □ 板上 MFCC + TFLM 推理 → 印 label
    • □ label → hand_control_execute() → 5 路 PWM pattern
    • □ 端對端：說 "Grasp" → 手合；"Release" → 手開 + WM8978 回 ACK 音
  6. 階段 6 — Demo 彩排（M8）
    • □ LCD 顯示 + 節能計數
    • □ 20 次彩排（走火場景 + 雜訊干擾測試）
    • □ 降級方案備妥（SW1 按下 = 硬碼動作）

8.3 風險管理 + Mitigation

風險機率影響降級 / 緩解
